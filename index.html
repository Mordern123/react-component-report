開發環境建置

若寫一個 React 的 SPA 網頁，依賴各種第三方 library 來處理各種基本功能。

  • Babel： 編譯 ES6+ 讓瀏覽器看懂
  • Webpack：打包資源與程式碼
  • eslint：檢查程式碼

Babel:

Babel是一個 JavaScript 編譯器，用於將 ES6+ 語法轉換為向後兼容的版本。 讓使用者可以使用最新版本的 JavaScript，同時保持瀏覽器的相容性。 將這些新特性轉換為瀏覽器可以理解的 ES5 語法，實現編譯和部署。

Webpack:

Webpack可以將前端應用程式中的所有資源（例如JavaScript、CSS、圖片等）作為模塊進行處理， 然後將它們打包成一個或多個較小的bundle，這些bundle可以在網頁上載入。 就是幫我們編譯我們的Preprocess成瀏覽器看得懂的內容然後打包成一包的完成檔案然後拿去server 上傳上去。

eslint:

ESLint是一個JavaScript的linting工具，它可以靜態分析JavaScript代碼，並檢查其中的錯誤同時強制實施程式碼風格， 以確保代碼的一致性和可讀性，如：語法或是 coding style。

開發環境建置

每個套件背後都是一個全新的世界，但現在各種 React 專案環境的需求大同小異，要能編譯，要能打包，要能檢查 lint，只有商業邏輯的差異比較大，那既然大家都差不多，怎麼沒有一個「模板」可以直接一鍵生成呢？

最初的靈感來自 XHP，由Facebook發布的JS library。
一開始，class component 的寫法是 React 的主流(舊版官方文件、歷史較久的專案)。

React 更新官方文件

改為使用Functional Component作為主要介紹的方法。

為什麼? Class 本來就能做到呀!!!

  • 編譯速度較快
  • 相依性較低
    • ● Functional Component 被設計為無狀態，需要管理狀態則掛勾於 Hooks 上，因此也無需使用到 this。
    ● Class Component 可以擁有自己的狀態，並起依賴於其他元件，設計上更為複雜，需要用到 this。
  • 可測試性及重用性高
    • ● 除了上述提到沒有自己內部狀態外，也沒有自己的狀態跟生命週期，測試更加簡易。
    ● 同時 Functional Component 常常被認為是 pure function，給相同 props，始終會回傳相同結果。這使得他更容易重用。
  • 較為簡單、易於理解、降低開發成本
  • 更多的社群資源

Class 表示：我也不是一無是處!

撰寫上遇到的問題

沒有建構子竟然也可以拿到 props!

為什麼要寫 super(props)?

在 JavaScript 中，super 會參照父類別的建構子。

並且在呼叫父類別的建構子後，才能在建構子中使用 this

              
                class TestComponent extends React.Component {
                  constructor(props) {
                    // 🔴 這裡還不能用this
                    super(props);
                    // ✅ 這裡可以用了!!!
                    this.state = { isOpen: true };
                  }
                }
              
            
              
                class FatherComponent {
                  constructor(name) {
                    this.name = name;
                  }
                }
                
                class SonComponent extends FatherComponent {
                  constructor(name) {
                    this.getSonName(); // 🔴 this.name還沒被定義
                    super(name);
                  }
                  getSonName() {
                    alert('My name is Eric!');
                  }
                }

為什麼要帶入props?

帶入props到super是必須的，因為底層的React.Component建構子才能初始化this.props?

測時結果，竟然可以讀取到this.props!!!

事實上 在 React 呼叫你的建構子後，它也會馬上配置 props 到實例中

React 刻意設計的~

可以只寫 super() 而不用寫 super(props) 嗎？

NO!!

React 的確會在你的建構子執行之後自行配置 this.props

但是 this.props 從呼叫 super 到建構子的結尾仍然會是undefined 

            
              constructor(props) {
                super();
                console.log(this.props); // 🔴 undefined
              }
